## 2. 项目背景与目标

### 2.1 背景
当前 lsfMonitor 已具备采样、展示、分析能力，但存在以下现实问题：
1. 运行环境以 CentOS 7.9 为主，少量 Rocky 8.10，环境约束强。
2. 用户角色分化明显：工程师强调实时操作；管理员强调趋势、治理和报表。
3. Web 在 CentOS 7.9 环境体验较差，不适合作为核心路径。
4. 现有代码结构偏单体，长期维护和扩展成本上升。

### 2.2 目标
建设一个面向 IC/EDA 的 LSF 双角色系统：
- 工程师：Linux 命令行直接使用，快速查看作业与 memory 状态并获得参数建议。
- 管理员：双入口（CLI + 本地桌面 GUI），完成短中长期资源分析与报表输出。
- 平台层面：提升稳定性、可维护性和可测试性，支持渐进式升级。

### 2.3 非目标（本期不做）
- 不将 Web 作为主交互入口。
- 不做云原生微服务化改造。
- 不做跨地域多租户平台化建设。

## 3. 关键约束与边界（重点标注）

## 【关键约束 A：运行环境】
- 主环境：CentOS 7.9
- 次环境：Rocky 8.10
- 必须保证 Linux 下可直接运行并得到结果。

## 【关键约束 B：交互形态】
- 工程师主入口：CLI（必须）
- 管理员双入口：CLI + 本地 GUI（必须）
- Web 非主路径（可选，不进入本期核心范围）

## 【关键约束 C：业务场景】
- 面向 IC/EDA 研发场景，LSF 操作系统绑定强。
- 项目价值重点：从“数据展示”升级到“可执行的资源优化建议”。

## 【边界范围】
- In Scope：采集、分析、CLI、管理员 GUI、报表导出、测试体系。
- Out of Scope：Web 主平台、大规模分布式数据库替换、SaaS 化改造。

## 4. 用户角色与核心诉求

### 4.1 工程师角色
关注点：
- 作业状态（RUN/PEND/DONE/EXIT）
- 本人 memory 使用与预留合理性
- 提交参数建议（特别是 rusage_mem）

核心问题：
- 我的作业为什么慢/卡？
- 预留内存是否过高（浪费）或过低（失败风险）？
- 下次提交建议值是多少？

### 4.2 管理员角色
关注点：
- 短期资源状态（小时/日）
- 长期资源趋势（周/月/季度）
- 治理分析（队列拥塞、资源浪费、失败模式、license 冲突）

核心问题：
- 哪些队列长期拥塞？
- 哪些用户/项目资源利用低效？
- 是否需要容量调整或策略优化？

## 5. 功能需求（FR）

### 5.1 工程师端（CLI）
- FR-E1：作业状态查询（按 job/user/queue 过滤）
- FR-E2：个人 memory 分析（申请值/当前值/峰值/历史对比）
- FR-E3：参数建议（基于历史模式给出建议区间）
- FR-E4：个人趋势（7/30/90 天）

建议命令原型（2026-03-04 已确认口径）：
- `bmon jobs [uid]`（不传 uid 默认当前用户）
- `bmon mem [uid] --days 7`
- `bmon advise --job <jobid>`

### 5.2 管理员端（CLI）
- FR-M1：资源总览（队列/主机/用户/项目）
- FR-M2：短中长期趋势分析
- FR-M3：治理洞察（浪费排行/失败原因/冲突时段）
- FR-M4：日报/周报/月报导出

建议命令原型：
- `bmon mgmt overview --range 7d`
- `bmon mgmt trend --range 90d`
- `bmon report weekly --export csv,md`

### 5.3 管理员端（本地 GUI，非 Web）
- FR-G1：多页签总览（总览/队列/主机/用户/项目/License/异常）
- FR-G2：交互筛选（时间、队列、项目、用户等）
- FR-G3：图表+表格同屏分析
- FR-G4：一键导出 PNG/CSV/Markdown/PDF

## 6. 数据与指标需求

### 6.1 数据域
- 作业域：状态、提交参数、资源申请与实际使用、退出码
- 主机域：load、可用资源、利用率
- 队列域：RUN/PEND/SLOT、等待时长
- 用户/项目域：占用、效率、趋势
- License 域：使用量、冲突、过期风险

### 6.2 核心指标（统一口径）
1. 作业成功率
2. Memory 浪费率
3. 内存相关失败占比
4. 队列平均等待时长
5. Slot 利用率
6. CPU 利用率
7. Memory 利用率
8. License 冲突指数（可选）

---

## 7. 非功能需求（NFR）
- NFR-1 兼容性：CentOS 7.9 必须稳定；Rocky 8.10 兼容。
- NFR-2 可用性：常用 CLI 查询秒级响应。
- NFR-3 稳定性：采样失败可重试；异常不影响核心链路。
- NFR-4 可维护性：模块化、日志可追踪、关键流程可测试。
- NFR-5 安全性：敏感信息最小暴露，管理视图具备权限边界扩展能力。

## 9. 风险与应对
1. CentOS 7 依赖风险 → 采用分层依赖和降级运行模式。
2. 历史数据口径不一致 → 提前冻结口径并提供迁移脚本。
3. 双角色需求冲突 → 统一数据底座，前端分入口。
4. 需求膨胀延期 → 严格 MVP 收敛与里程碑验收。

---

## 10. 验收标准（高层）
- 工程师：能在 Linux CLI 快速获取状态、memory 分析和建议。
- 管理员：可完成短中长期分析并输出可汇报报表。
- 平台：CentOS 7.9/Rocky 8.10 均可部署，采集与分析链路稳定。

## 11. 已确认要点（来自需求讨论）
- 工程师使用命令行是主方案。
- 管理员需要双入口：命令行 + 更好的界面展示窗口。
- 项目不以 Web 为主路径。
- 管理员必须可查看长期与短期资源状态。
- 项目面向 IC 设计开发 LSF 场景，OS 绑定较严重。
- **重点新增：必须构建 CI/CD 模式，编码后的每一次提交都要完成编码质量检查 / 构建验证 / 测试 / 部署验证（至少 Dry-Run）。**

### 11.1 2026-03-04 新增确认（工程师体验专项）
1. 工程师 CLI 命令统一从 `lsfmon` 切换为 `bmon`，当前阶段不做兼容等待期。
2. `my` 子命令层级简化：
   - `bmon jobs` 默认查询当前用户
   - `bmon jobs <uid>` 查询指定用户
   - `bmon mem` / `bmon mem <uid>` 同理
3. `jobs` 输出增强：增加 `req_cor`、`req_mem`、`mem`、`run_time`；
   原 `ava_cpus` 需求按讨论口径落地为 `idle_factor = cputime/runtime`（可附带 `cpu_util%`）。
4. 安装后需提供双 shell 环境接入文件：
   - `monitor/conf/lsfmonitor.source.sh`（bash）
   - `monitor/conf/lsfmonitor.source.csh`（csh）
